## Introduction

Modern conversational AI is predominantly hosted in the cloud. While cloud APIs are convenient, they present issues with data privacy, platform dependency, and latency. **MOHINI AI** is an engineering project aimed at building a fully local, voice-first desktop companion.

In this deep dive, we'll examine the design decisions, trade-offs, and pipelines required to make speech interaction feel near-instantaneous (under 300ms total loop latency).

## 1. The Speech Pipeline

Making a local voice assistant feel conversational requires optimizing three consecutive tasks: Speech-to-Text (STT), LLM Inference, and Text-to-Speech (TTS).

### Voice Ingestion & Wake-word Detection

Rather than streaming continuous audio to a transcription engine, MOHINI uses a two-tier voice activation strategy:
1. **Low-power detection**: A lightweight wake-word engine (e.g. Porcupine) running locally.
2. **Dynamic thresholding**: A local RMS meter that begins recording when voice signals exceed the ambient noise floor.

<Callout type="info" title="System Requirement">
The audio input stream is configured at **16kHz, mono, 16-bit PCM**, matching the native input requirements of the whisper neural networks.
</Callout>

### Transcription Optimization

For STT, we benchmarked vanilla OpenAI Whisper vs. `faster-whisper` (a C++ implementation of Whisper using CTranslate2).

<EngineeringNote type="performance" title="CTranslate2 Optimization">
Using FP16 quantization on a GPU or INT8 on CPU via `faster-whisper` reduces transcription latency from **1.2s** down to **180ms** for average-length sentences.
</EngineeringNote>

Here is the initialization pipeline in Python:

```python
from faster_whisper import WhisperModel

# Initialize the model using INT8 quantization for CPU execution
model_size = "base.en"
model = WhisperModel(model_size, device="cpu", compute_type="int8")

def transcribe(audio_path):
    segments, info = model.transcribe(audio_path, beam_size=5)
    return " ".join([segment.text for segment in segments])
```

## 2. System Architecture & Decisions

The core execution loop of MOHINI relies on an autonomous Orchestration Engine. Unlike simple chat interfaces, MOHINI maps user intent to local OS tools.

<EngineeringNote type="decision" title="Orchestrator Selection: LangChain vs. Custom Loop">
We selected a custom, event-driven state loop over LangChain. While LangChain is feature-rich, its nested abstractions added **150ms** of overhead to each prompt execution. A bare-metal Python dictionary dispatcher reduced loop routing to **&lt;2ms**.
</EngineeringNote>

## 3. Local LLM Quantification Benchmarks

Running Llama 3 on consumer-grade hardware requires quantizing model weights to fit within VRAM boundaries. Below is a trade-off comparison of various quantization configurations:

| Quantization Format | Model Size | VRAM Usage | Latency (tokens/s) | Perplexity (Lower is Better) |
| :--- | :--- | :--- | :--- | :--- |
| **FP16 (Unquantized)** | 16.0 GB | ~18.0 GB | 12 t/s | 5.12 |
| **Q8_0 (8-bit)** | 8.5 GB | ~9.5 GB | 28 t/s | 5.15 |
| **Q4_K_M (4-bit)** | 4.8 GB | ~5.8 GB | 54 t/s | 5.23 |
| **Q2_K (2-bit)** | 2.9 GB | ~3.8 GB | 72 t/s | 6.45 |

<EngineeringNote type="tradeoff" title="Model Quality vs. Speed">
We selected **Q4_K_M** (4-bit quantization). It offers a **4.5x** increase in inference speed with negligible perplexity degradation, allowing the model to run comfortably on GPUs with 6GB VRAM.
</EngineeringNote>

## 4. Execution Guardrails

Because MOHINI is capable of executing command-line scripts directly on the host OS (e.g., opening directories, killing processes, searching files), we implemented strict security guardrails.
